# 真实硬件接入方案

> 本文为 SleepCare 项目设计真实睡眠监测硬件接入方案。核心原则：**表结构不变、API 不变，只替换数据写入层**。

---

## 一、数据链路改造

### 1.1 现状

当前系统通过 `backend/utils/generator.js`（或类似模块）每日生成模拟睡眠数据，直接写入 `sleep_reports` 表。数据维度包括：

- 睡眠评分、总时长、深睡/浅睡/REM 时长
- 心率曲线、睡眠分期、噪音数据

### 1.2 真实硬件数据流

```
睡眠监测硬件
    ↓（MQTT / HTTP）
云端 Broker / 后端 API
    ↓
sleep_reports 表（表结构不变）
    ↓
小程序 / 医生端 Web 展示
```

### 1.3 改造要点

| 项目 | 模拟数据阶段 | 真实硬件阶段 |
|------|-------------|-------------|
| 数据来源 | 代码生成 | 硬件传感器上报 |
| 上报协议 | 无 | MQTT over TLS 或 HTTPS |
| 写入入口 | 定时任务 | 设备网关 → 后端数据写入服务 |
| 表结构 | `sleep_reports` | **不变** |
| API | 现有接口 | **不变** |

### 1.4 数据采集频率

- 心率：每 1 分钟采样一次
- 体动/呼吸：每 10 分钟聚合一次
- 睡眠分期：每 5 分钟计算一次阶段
- 噪音：每 10 分钟记录一次平均值

后端接收原始数据后，按现有 `sleep_reports` 字段格式进行聚合、清洗、入库。

## 二、设备绑定与指令下发

### 2.1 用户流程变化

**当前流程**：
1. 用户手动输入 16 位序列号
2. 后端创建虚拟设备记录

**真实硬件流程**：
1. 用户首次开启硬件，进入配网模式
2. 小程序通过蓝牙/Wi-Fi 扫描发现设备
3. 用户选择 Wi-Fi 并输入密码
4. 硬件联网后向后端注册设备
5. 后端写入 `devices` 表，`online_status = 1`

### 2.2 设备在线状态维护

- 硬件每 30 秒发送一次心跳包
- 后端收到心跳更新 `devices.online_status = 1` 和 `updated_at`
- 超过 2 分钟未收到心跳，后端将 `online_status` 置为 0
- 用户小程序首页显示设备在线/离线状态

### 2.3 指令下发（OTA / 配置）

- 时区设置、勿扰时段通过 MQTT topic 下发
- 硬件订阅 `device/{serial_no}/command`
- 执行完成后上报 ack

## 三、数据库迁移方案

### 3.1 从 SQLite 迁移到 MySQL

开发阶段使用 SQLite，生产环境使用 MySQL。

迁移步骤：

1. 执行 `docs/migration.sql` 在 MySQL 中创建表结构
2. 运行 `docs/migrate-data.js` 将 SQLite 数据导入 MySQL
3. 设置环境变量 `DATABASE_TYPE=mysql` 启动后端

### 3.2 迁移后验证

- 用户登录正常
- 设备列表正常
- 睡眠报告查询正常
- 医生授权与建议正常

### 3.3 回滚策略

- 迁移前备份 SQLite 文件
- 迁移脚本幂等：可先清空 MySQL 表再导入
- 业务双写过渡期：SQLite 与 MySQL 同时写入，验证一致后再切流

## 四、部署与运维

### 4.1 生产部署架构

```
用户小程序
    ↓ HTTPS
Nginx（反向代理 + 负载均衡）
    ↓
Express 后端服务集群（PM2 管理）
    ↓
MySQL 主从 + Redis 缓存
    ↓
MQTT Broker（EMQX / Mosquitto）
    ↓
睡眠监测硬件
```

### 4.2 数据库连接池配置

- MySQL 连接池大小：10 ~ 50
- 连接超时：30 秒
- 重试策略：连接失败时 3 次指数退避重试

### 4.3 数据归档策略

- `sleep_reports` 热数据保留 1 年
- 超过 1 年的报告迁移到归档表或对象存储
- 归档数据仍可通过 API 查询（查询归档库）

### 4.4 监控告警

| 监控项 | 告警阈值 | 处理方式 |
|--------|---------|----------|
| 硬件离线 | 5 分钟未收到心跳 | 推送用户通知 |
| 数据异常 | 连续 3 小时无报告 | 触发人工排查 |
| API 错误率 | > 5% | 告警并自动重启 |
| MySQL 连接数 | > 80% | 扩容连接池 |

## 五、实施路径

| 阶段 | 目标 | 周期 |
|------|------|------|
| Phase 1 | 保持现有架构，硬件通过 HTTP API 写入 `sleep_reports` | 2 周 |
| Phase 2 | 引入 MQTT Broker，后端消费硬件消息写入数据库 | 2 周 |
| Phase 3 | 引入时序数据库（InfluxDB）存储高频数据，MySQL 保留业务数据 | 4 周 |

---

## 六、总结

真实硬件接入的关键是：**保持业务表结构和 API 不变，在数据写入层引入硬件网关**。通过 MQTT/HTTP 接收硬件数据，经过聚合后写入 `sleep_reports`，即可让现有小程序和医生端 Web 无感知地支持真实设备。
