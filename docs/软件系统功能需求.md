# 智能睡眠环境调控设备 - 软件系统功能需求文档（实训需求）

> 版本：V1.0 ｜ 日期：2026年6月15日
> 开发范围：微信小程序 + 云后台API + 医生端Web后台
> 依赖说明：不依赖真实硬件设备，所有设备数据、环境调控指令均使用模拟数据或预留接口。硬件就绪后，仅需替换数据源和控制指令下发模块即可无缝接入。

---

## 一、项目背景

根据"智能睡眠环境调控设备"的完整软硬件系统。其中，软件部分包括微信小程序（用户端）和云端管理后台（含医生端Web）。

为降低硬件依赖带来的开发风险，并提前验证用户体验、数据管理及医疗协同流程，本阶段先独立开发与硬件无关的软件系统，待提供真实硬件模块后，再通过预留接口对接实际数据采集与控制指令。

### 软件系统总体架构

系统分为两层：

- **上层**：用户直接交互的微信小程序，包含全部业务功能。
- **下层**：云端服务提供API接口、模拟数据生成、数据持久化、固件管理以及医生专用Web后台。

### 无硬件实现方式

| 维度 | 实现方式 |
|------|---------|
| 设备绑定 | 用户可添加"虚拟设备"（后台自动生成设备ID），或手动输入设备序列号（不校验真实在线状态）。 |
| 睡眠数据 | 后台内置模拟数据生成器，根据用户设定的作息时间，每日自动生成符合生理规律的模拟睡眠数据。 |
| 环境调控 | 小程序中的手动控制面板界面保留，但任何调节操作均弹窗提示"硬件未就绪"；自动调控功能（如自动熄灯）不在本阶段实现。 |
| OTA升级 | 小程序展示固件版本和"检查更新"按钮，点击后获取模拟版本信息，不执行实际升级。 |

## 三、微信小程序功能需求

### 3.1 用户账号体系

| 功能点 | 描述 |
|--------|------|
| 注册 | 支持手机号+验证码注册，或微信一键授权（获取昵称、头像）。 |
| 登录 | 手机号/微信授权登录，Token自动维护。 |
| 个人信息 | 修改昵称、头像、性别、出生年份。 |
| 账号注销 | 符合个保法，提供注销功能（删除所有个人数据，并解除所有设备绑定）。 |

### 3.2 设备管理（抽象层）

| 功能点 | 描述 |
|--------|------|
| 添加设备 | 方式1：手动输入16位设备序列号（后台校验格式，不校验真实存在）；方式2：点击"添加演示设备"，后台生成虚拟设备并绑定。 |
| 设备列表 | 展示已绑定的设备（可编辑昵称、查看设备型号和固件版本）。 |
| 解绑设备 | 解绑后该设备产生的历史数据仍保留在用户账户下（不可见但后台保留）。 |
| 在线状态 | 虚拟设备的在线状态默认显示"在线"，用户可手动切换为"离线"用于测试。 |

### 3.3 睡眠报告（基于模拟数据）

**基础版报告**：首页卡片展示最近一晚的睡眠评分（0-100）、总睡眠时长、深睡比例、觉醒次数。

**完整版报告**：提供详细报告页面，包含：
- 睡眠阶段曲线（深睡、浅睡、REM、清醒），以折线图或堆叠条形图展示。
- 心率趋势图（夜间每分钟均值）。
- 呼吸频率趋势图。
- 环境噪音等级曲线（模拟数据）。
- 异常事件列表（如"呼吸暂停可能""频繁觉醒"，由模拟数据随机生成标注）。
- 支持按日、周、月查看历史报告，并提供趋势图（如近7天睡眠评分走势）。

**报告分享**：
- 生成睡眠报告截图（图片）或PDF，可通过微信分享给好友/家人。
- 支持一键导出CSV数据（匿名，不含个人身份信息）。

### 3.4 e-CBTi睡眠训练专区

该专区作为小程序的一个独立一级入口，必须包含以下四个子模块：

| 功能点 | 描述 |
|--------|------|
| 睡眠卫生教育 | 图文静态展示睡眠卫生知识（由甲方提供内容，乙方负责排版和展示）。支持多篇文章分类、收藏。 |
| 睡眠日记 | 用户每天可填写主观睡眠日记，至少包含：上床时间、入睡时间、夜间觉醒次数、起床时间、日间困倦程度（1-5分）。支持日历视图查看历史记录。数据存储于云端。 |
| 专业测评 | 内置 ISI（失眠严重程度指数量表7题）、PSQI（匹兹堡睡眠质量指数24题）、ESS（Epworth嗜睡量表8题）。每个量表完成后自动计算得分并给出等级评定。历史测评结果可查看。 |
| 睡眠限制/刺激控制打卡 | 用户可设定就寝时间窗和起床时间窗。每日打卡：记录实际就寝时间、实际起床时间。系统基于预设规则（如实际卧床时间>8小时）给出弹窗提示（如"请缩短卧床时间"）。提示规则由后台配置。 |

### 3.5 医生授权管理

| 功能点 | 描述 |
|--------|------|
| 授权医生 | 用户输入医生手机号或医生ID，发起授权请求。系统向医生发送短信/小程序订阅消息通知。 |
| 授权管理 | 查看已授权的医生列表，显示授权剩余天数，可随时撤销授权。 |
| 授权有效期 | 默认30天，用户可自定义（7/15/30/60天）。到期自动失效，用户可续期。 |
| 数据范围 | 授权后医生可查看该用户的睡眠报告（完整版）、e-CBTi睡眠日记、测评结果、打卡记录。不显示真实姓名，仅显示匿名ID。 |

### 3.6 消息通知

- 每日早上推送"睡眠报告已生成"提醒（微信订阅消息）。
- 当虚拟设备或真实设备离线超过24小时，推送提醒（可后台模拟触发）。
- 医生授权/撤销/到期提醒。
- 夜间勿扰时段内（根据用户设置）不推送任何通知。

### 3.7 OTA固件升级（混合升级策略）

#### 3.7.1 设计原则
- **用户知情**：用户有权知道固件更新内容，并选择安装时机。
- **避免打扰**：自动下载不打断用户使用，安装需用户确认或安排在非睡眠时段。
- **安全性**：关键安全更新可强制提醒，但用户仍可推迟（有限次数）。

#### 3.7.2 混合升级流程

| 步骤 | 说明 |
|------|------|
| 1. 后台发布新固件 | 管理员在云后台上传固件文件（.bin），填写版本号、更新日志、灰度比例。 |
| 2. 设备自动检测与下载 | 设备（真实硬件）定期轮询云端，发现新固件后自动下载到本地存储（不安装）。 |
| 3. 用户通知 | 下载完成后，小程序首页或设备详情页显示"新固件已就绪"红点提醒。 |
| 4. 用户确认安装 | 用户点击"立即更新"，设备在空闲时（非睡眠时段）执行安装。 |
| 5. 夜间避开逻辑 | 若用户设定的睡眠时段（如23:00-7:00）内不会自动推送安装提醒；用户主动点击更新时，需二次确认"当前为睡眠时段，是否继续？" |
| 6. 强制升级 | 对于重要安全更新，后台可标记"强制"，用户只能选择"稍后提醒"（最多3次）或立即安装。 |

#### 3.7.3 无硬件依赖版中的实现（仅UI与模拟）
- 设备详情页：显示当前固件版本（模拟数据，如"V1.0.0"）。
- 检查更新按钮：点击后调用 `/device/firmware/latest` API，返回模拟的版本号（如"V1.2.0"）和更新日志。
- 弹窗提示："发现新版本 V1.2.0（演示模式，无实际升级）"，不执行任何真实下载或安装。
- 红点提醒：可模拟后台随机触发"有新固件"的状态（通过API返回 has_update=true）。
- 后续硬件接入时：替换为真实的OTA流程（调用设备端升级接口）。

#### 3.7.4 与后台自动升级的关系
- 手机端入口作为用户主动触发的补充。
- 后台自动下载功能由设备端固件实现，云端仅负责提供固件文件及版本信息。本阶段无需开发设备端自动下载逻辑。

### 3.8 个性化设置模块（完整版）

该模块集中管理用户的所有偏好配置，所有设置均存储于云端，供后续硬件接入时同步。

#### 3.8.1 作息计划

| 设置项 | 说明 |
|--------|------|
| 期望就寝时间 | 格式 HH:MM，用于模拟数据生成及后续唤醒定时 |
| 期望起床时间 | 格式 HH:MM，用于模拟数据生成及后续唤醒定时 |
| 日出程序时长 | 5-30分钟可调，默认10分钟（仅存储） |

#### 3.8.2 助眠偏好

| 设置项 | 说明 |
|--------|------|
| 助眠音效类型 | 白噪音/海浪/雨声/篝火等，可试听（本地播放音频文件） |
| 唤醒声音类型 | 鸟鸣/轻柔音乐/自然音，可试听 |

#### 3.8.3 环境设置（模拟）

| 设置项 | 说明 |
|--------|------|
| 偏好亮度等级 | 1-100（对应1-300 lux），仅存储，无实际硬件控制 |
| 偏好音量等级 | 0-100（对应静音-85dB），仅存储 |
| 手动控制面板 | 保留亮度/色温/音量滑块，任何调节操作弹窗"演示模式，无实际硬件" |

#### 3.8.4 设备设置

| 设置项 | 说明 |
|--------|------|
| 设备时区 | 默认自动获取手机时区，可手动修改（影响报告日期显示和模拟数据生成时间） |
| 夜间勿扰模式 | 开关，设置起止时间（如23:00-6:00） |
| 勿扰时段行为 | 不推送任何通知；硬件接入后可同时降低灯光亮度或静音。 |

### 3.9 UI/UX商业级要求

整体视觉风格符合主流医疗健康App规范（柔和色调、圆角卡片、清晰层级）。

## 四、云后台功能需求

### 4.1 基础API服务

- **用户管理API**：注册、登录（JWT）、获取/修改个人信息、注销。
- **设备管理API**：添加设备（手动输入序列号或生成虚拟设备）、解绑、获取设备列表、更新设备昵称。虚拟设备自动生成唯一序列号（格式：VIR+16位随机字符），标记 is_virtual=1。
- **睡眠报告API**：接收模拟数据生成器产生的数据，提供按日/周/月查询报告摘要及详细数据的接口。
- **个性化设置API**：获取/修改作息计划、助眠偏好、环境设置、设备设置。
- **医生授权API**：医生端认证（手机号+密码/验证码）；患者发起授权、医生查询授权请求、同意/拒绝、撤销授权、获取授权患者列表；医生获取患者睡眠数据（只读、脱敏）。
- **日志与监控**：API请求日志、错误告警（邮件/钉钉）；设备在线状态模拟：虚拟设备默认在线，可后台手动批量设置离线。

### 4.2 睡眠数据模拟生成器（后台内部工具）

**功能**：定时任务（每日凌晨2点）为每个绑定了虚拟设备且有作息设置的用户生成前一晚的模拟睡眠数据。

**生成规则**：
- 总睡眠时长 = 用户设定的就寝时间到起床时间之间的80%~95%（随机）。
- 深睡占比15%~25%，REM占比20%~25%，浅睡占比45%~55%。
- 心率曲线：入睡阶段75bpm → 深睡最低55bpm → 快速眼动期回升至70bpm → 清晨觉醒前升至80bpm。
- 呼吸频率：16次/分 → 12次/分 → 16次/分。
- 觉醒次数：0~3次，每次清醒2-5分钟。
- 异常事件：随机生成"呼吸暂停可能"或"周期性肢体运动"标签，概率5%。
- 手动触发：管理员可通过后台API为指定用户强制生成某天的数据。

### 4.3 e-CBTi数据支持与接口

- 数据库扩展表（详见第五节）。
- API接口：保存睡眠日记、查询历史日记；提交量表答案并返回得分；保存打卡记录、获取打卡日历；获取系统预设的提示规则（后台可配置，如"卧床时间>9小时则提示缩短卧床时间"）。

### 4.4 OTA固件管理（后台）

**功能**：
- 固件版本管理：上传固件程序、版本号、更新日志、灰度发布比例。
- 设备升级任务跟踪：记录每台设备的升级状态（待下载/已下载/已安装/失败）。
- 强制升级标记：可标记某版本为"强制升级"，设备端需特殊处理。

**API供小程序调用**：
- `/device/firmware/latest`：返回最新固件版本信息（含是否强制、更新日志）。
- `/device/firmware/update`：用户确认安装时调用，记录用户同意时间（无硬件阶段仅返回模拟成功）。

**无硬件依赖版实现**：
- 所有API返回模拟数据（如版本号"V1.2.0"，has_update=true，is_forced=false）。
- 不需要实际存储固件文件，但预留上传接口和数据库表结构。

### 4.5 医生端Web后台

**功能需求**：
- 医生登录/注销（手机号+验证码或密码）。
- 首页展示授权患者列表（匿名ID、最近睡眠评分、授权剩余天数、最新睡眠日记摘要）。
- 点击患者进入详情页，可查看：睡眠报告（同小程序完整版图表）；睡眠日记日历及详细记录；量表测评历史（ISI/PSQI/ESS得分趋势）；睡眠限制打卡记录。
- 医生可在患者详情页下方添加文字建议（干预意见），患者在小程序内可查看。
- 日志审计：记录每位医生查看患者数据的时间、患者ID。
- 技术：响应式Web页面，适配PC端和Pad端。

### 4.6 其他管理功能

- 用户管理（运营）：查看用户列表、禁用/启用账号、导出脱敏数据。
- 设备管理：查看所有设备（虚拟/真实），手动触发模拟数据生成，开关全局模拟数据生成器。

## 五、数据结构设计（核心表）

### 5.1 用户表 users

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| user_id | bigint | PRIMARY KEY, AUTO_INCREMENT | 用户唯一标识，系统自动生成 |
| phone | varchar(20) | NOT NULL, UNIQUE | 手机号码，用于登录和身份识别，格式为11位数字 |
| nickname | varchar(50) | NOT NULL, DEFAULT '用户' | 用户昵称，可修改，最长50字符 |
| avatar_url | varchar(255) | NULL | 头像图片的URL地址，可为空，默认使用系统头像 |
| gender | tinyint | NOT NULL, DEFAULT 0 | 性别：0=未知，1=男，2=女 |
| birth_year | int | NULL | 出生年份（4位数字），用于年龄计算和健康分析 |
| role | tinyint | NOT NULL, DEFAULT 0 | 用户角色：0=普通用户，1=医生，2=管理员 |
| status | tinyint | NOT NULL, DEFAULT 0 | 账户状态：0=正常，1=禁用（禁用后无法登录） |
| created_at | datetime | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 账户注册时间 |

### 5.2 设备表 devices

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| device_id | varchar(32) | PRIMARY KEY | 设备唯一序列号，硬件设备为出厂编码，虚拟设备由系统生成（格式：VIR+16位随机字符） |
| user_id | bigint | FOREIGN KEY (users.user_id), NULL | 当前绑定用户的ID，NULL表示设备未绑定 |
| device_name | varchar(50) | NOT NULL, DEFAULT '我的设备' | 用户自定义的设备昵称，用于在列表中显示 |
| is_virtual | tinyint | NOT NULL, DEFAULT 0 | 是否为虚拟设备：1=虚拟（用于演示/模拟），0=真实硬件 |
| firmware_version | varchar(20) | NOT NULL, DEFAULT 'V1.0.0' | 设备当前固件版本号，格式如 V1.0.0 |
| last_active_time | datetime | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 设备最后活跃时间（真实设备上报心跳），虚拟设备每日自动更新 |
| created_at | datetime | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 设备首次添加到系统的时间 |

### 5.3 睡眠报告表 sleep_reports

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| report_id | bigint | PRIMARY KEY, AUTO_INCREMENT | 睡眠报告唯一标识 |
| user_id | bigint | FOREIGN KEY (users.user_id), NOT NULL | 所属用户ID |
| device_id | varchar(32) | FOREIGN KEY (devices.device_id), NOT NULL | 产生此报告的设备ID |
| report_date | date | NOT NULL | 报告对应的夜晚日期，与user_id组成唯一索引 |
| sleep_score | int | NOT NULL, DEFAULT 0 | 睡眠综合评分，0-100分 |
| total_minutes | int | NOT NULL, DEFAULT 0 | 总睡眠时长（分钟），包含深睡+浅睡+REM |
| deep_minutes | int | NOT NULL, DEFAULT 0 | 深睡时长（分钟），正常成人占比15%-25% |
| rem_minutes | int | NOT NULL, DEFAULT 0 | 快速眼动睡眠时长（分钟），正常成人占比20%-25% |
| light_minutes | int | NOT NULL, DEFAULT 0 | 浅睡时长（分钟），正常成人占比45%-55% |
| wake_minutes | int | NOT NULL, DEFAULT 0 | 夜间总清醒时长（分钟） |
| avg_heart_rate | decimal(5,2) | NULL | 整夜平均心率（次/分钟），正常静息心率范围40-100 |
| events_json | text | NULL | 异常事件列表，JSON数组格式 |
| heart_rate_curve | text | NULL | 分钟级心率数据，JSON数组 |
| respiration_curve | text | NULL | 分钟级呼吸频率数据，JSON数组 |
| stage_curve | text | NULL | 睡眠分期序列，每30秒一个值：0=清醒，1=浅睡，2=深睡，3=REM |
| noise_curve | text | NULL | 环境噪音等级曲线（仅模拟），每分钟一个数值，单位dB SPL |

## 六、API接口清单（无硬件依赖版）

所有API均返回统一格式 `{code, message, data}`：
- **code**：整数，状态码。0表示成功，其他数字表示各类错误（1001参数错误，1002未登录等）。
- **message**：字符串，对 code 的文字说明。
- **data**：对象/数组/null，实际返回的业务数据。

**示例 — 登录成功**：
```json
{
    "code": 0,
    "message": "success",
    "data": {
        "user_id": 12345,
        "nickname": "睡眠用户",
        "token": "eyJhbGciOiJIUzI1NiIs..."
    }
}
```

**示例 — 参数错误**：
```json
{
    "code": 1001,
    "message": "手机号不能为空",
    "data": null
}
```

## 九、后续硬件接入时的改动点

当提供真实硬件后，仅需进行以下最小改动：

1. **数据源替换**：将睡眠报告的模拟数据生成器改为接收真实硬件上报的数据（通过MQTT或HTTP），数据结构保持一致。
2. **设备管理**：添加设备时，真实设备通过蓝牙配网写入真实序列号，不再生成虚拟设备。
3. **手动控制**：将控制面板的滑块点击事件改为通过MQTT下发真实指令。
4. **自动调控**：增加嵌入式端状态机，不在小程序侧实现。
5. **个性化设置同步**：硬件端读取用户已保存的作息计划、助眠偏好、环境设置等，作为设备初始化参数。
6. **OTA升级**：将小程序的模拟版本查询和确认安装逻辑替换为真实的OTA指令（调用设备端接口执行下载和安装）。后台自动下载功能由设备端固件实现，云端仅提供固件文件。

> 无需重构任何e-CBTi、医生授权、报告展示等业务逻辑。
