# 医患通知方案设计文档

> 本文档基于 Claude Plan 模式讨论产出，用于设计“医生填写干预建议后，患者如何收到通知并在小程序中查看”的完整方案。

---

## 一、问题分析

### 1.1 现状

- 医生在 Web 端查看患者报告后，可以在页面下方填写干预建议。
- 建议保存到 `doctor_authorizations.doctor_note` 字段。
- 但是：医生写了建议后，**患者完全不知道**，也不会主动在小程序中查看。

### 1.2 核心需求

| 角色 | 痛点 | 期望 |
|---|---|---|
| 医生 | 写了建议但患者可能看不到 | 建议能触达患者 |
| 患者 | 不知道医生给自己写了建议 | 进入小程序能有提示，并能看到建议内容 |

### 1.3 约束条件

- 本项目是实训项目，**无硬件依赖**。
- 不接入微信服务器推送（模板消息/订阅消息）配置较复杂，适合作为可选方案。
- 方案必须简单可行，适合在 1 节课内落地。

---

## 二、方案对比

### 方案一：患者主动查询（纯轮询）

**思路**：患者每次进入小程序或报告页时，主动调用 API 查询是否有医生建议。

**优点**：
- 实现最简单，无需额外服务。
- 不依赖微信模板消息、订阅消息等能力。

**缺点**：
- 不是实时通知，患者不进入小程序就看不到。
- 体验较差，没有“未读提示”。

**适用场景**：无推送能力、快速验证原型。

---

### 方案二：微信小程序订阅消息推送

**思路**：医生保存建议后，后端调用微信订阅消息 API，向患者推送“医生有新建议”通知。

**优点**：
- 真正的实时推送，体验好。
- 患者即使不打开小程序也能收到通知。

**缺点**：
- 需要配置微信小程序订阅消息模板。
- 需要患者在使用小程序时授权订阅。
- 需要维护微信 access_token、模板 ID 等，配置成本较高。
- 超出本次实训核心范围。

**适用场景**：生产环境、追求用户体验。

---

### 方案三：小程序后台轮询 + 红点提示（推荐方案）

**思路**：

1. 患者打开小程序时，自动调用后端接口检查是否有未读建议。
2. 如果有，在首页 tabBar 或“报告”页显示红点/数字角标。
3. 患者点击进入“医生建议”页面查看建议详情。
4. 查看后调用后端接口标记为“已读”，红点消失。

**优点**：
- 实现简单，不需要微信服务器推送配置。
- 有明显提示，比纯轮询体验好。
- 适合实训项目，1 节课可落地。

**缺点**：
- 不是实时推送，依赖患者打开小程序。
- 但比“纯轮询”好很多，因为进入首页即有红点提示。

**适用场景**：本次实训项目。

---

## 三、推荐方案：方案三（轮询 + 红点提示）

### 3.1 整体流程

```
医生保存建议
    ↓
后端写入 doctor_authorizations.doctor_note
    ↓
患者打开小程序 → 首页 onShow 调用 GET /api/patient/note/check
    ↓
hasNew=true → 首页/报告 tabBar 显示红点
    ↓
患者点击“医生建议”入口 → 进入列表页
    ↓
调用 GET /api/patient/note/list 获取建议列表
    ↓
点击查看详情 → 查看完整建议
    ↓
返回列表时，调用 PUT /api/patient/note/read 标记已读
    ↓
红点消失
```

### 3.2 数据库改动

在 `doctor_authorizations` 表增加 `patient_read_at` 字段，用于记录患者查看时间。

```sql
ALTER TABLE doctor_authorizations
ADD COLUMN patient_read_at TEXT;
```

**字段含义**：
- `NULL`：患者尚未查看。
- 非空时间字符串：患者已查看的时间。

### 3.3 后端 API 设计

#### 1. 检查是否有新建议

```http
GET /api/patient/note/check
Authorization: Bearer {token}
```

**响应**：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "hasNew": true,
    "count": 1
  }
}
```

**实现逻辑**：

```sql
SELECT COUNT(*) FROM doctor_authorizations
WHERE patient_id = ? AND status = 'active' AND doctor_note IS NOT NULL AND doctor_note != '' AND (patient_read_at IS NULL OR patient_read_at = '')
```

---

#### 2. 获取医生建议列表

```http
GET /api/patient/note/list
Authorization: Bearer {token}
```

**响应**：

```json
{
  "code": 0,
  "message": "success",
  "data": [
    {
      "id": 11,
      "doctor_id": 9,
      "doctor_name": "测试医生",
      "doctor_phone": "13800001111",
      "note": "建议保持规律作息，睡前避免使用手机。",
      "created_at": "2026-06-30 21:18:22",
      "is_read": false
    }
  ]
}
```

**实现逻辑**：

```sql
SELECT a.id, a.doctor_id, u.nickname, u.phone, a.doctor_note, a.updated_at, a.patient_read_at
FROM doctor_authorizations a
JOIN users u ON a.doctor_id = u.id
WHERE a.patient_id = ? AND a.status = 'active' AND a.doctor_note IS NOT NULL AND a.doctor_note != ''
ORDER BY a.updated_at DESC
```

---

#### 3. 标记建议为已读

```http
PUT /api/patient/note/read
Authorization: Bearer {token}
Content-Type: application/json

{
  "note_id": 11
}
```

**响应**：

```json
{
  "code": 0,
  "message": "已标记为已读",
  "data": null
}
```

**实现逻辑**：

```sql
UPDATE doctor_authorizations
SET patient_read_at = datetime('now','localtime')
WHERE id = ? AND patient_id = ? AND status = 'active'
```

---

### 3.4 小程序页面设计

#### 页面 1：首页红点提示

- 在 `pages/home/home.js` 的 `onShow` 中调用 `GET /api/patient/note/check`。
- 如果 `hasNew` 为 true，在首页右上角或“医生建议”入口显示红点。
- 可选：在 `app.json` 的 `tabBar` 的“报告”项上显示红点（小程序原生 tabBar 红点 API：`wx.showTabBarRedDot`）。

#### 页面 2：医生建议列表页

- 新建 `pages/doctor_notes/doctor_notes` 页面。
- 列表项显示：医生姓名、建议摘要、时间、是否已读。
- 点击列表项进入详情页。

#### 页面 3：建议详情页

- 新建 `pages/doctor_note_detail/doctor_note_detail` 页面。
- 展示完整建议内容。
- 进入页面后，调用 `PUT /api/patient/note/read` 标记已读。

---

### 3.5 与现有功能的关系

| 现有功能 | 与本方案的关系 |
|---|---|
| 医生端 Web 保存建议 | 触发点，保存后患者即可在小程序看到 |
| 患者授权医生 | 只有 active 授权的建议才会展示给患者 |
| `doctor_authorizations.doctor_note` | 已存在，直接复用 |
| `doctor_authorizations.patient_read_at` | 新增字段，用于判断红点 |

## 四、实施步骤

### 步骤 1：数据库改动

在 `backend/db/schemas.js` 中，`doctor_authorizations` 表增加字段：

```js
patient_read_at TEXT  // 患者查看时间，NULL=未读，非空=已读
```

### 步骤 2：后端新增 API

在 `backend/routes/` 下新建 `patient.js`（或扩展已有路由），实现：

- `GET /api/patient/note/check`
- `GET /api/patient/note/list`
- `PUT /api/patient/note/read`

在 `backend/app.js` 中挂载：`app.use('/api/patient', patientRoutes)`。

### 步骤 3：小程序页面

1. 新建 `pages/doctor_notes/doctor_notes`（列表页）。
2. 新建 `pages/doctor_note_detail/doctor_note_detail`（详情页）。
3. 在 `app.json` 注册这两个页面。
4. 在 `pages/home/home.js` 的 `onShow` 中调用 `/api/patient/note/check`，并显示红点。
5. 在首页增加“医生建议”入口卡片，带红点提示。

### 步骤 4：测试流程

1. 医生登录 Web → 查看患者报告 → 填写建议 → 保存。
2. 患者打开小程序 → 首页显示红点。
3. 患者点击“医生建议” → 看到列表 → 点击进入详情 → 红点消失。

---

## 五、方案取舍说明

| 方案 | 是否推荐 | 原因 |
|---|---|---|
| 方案一：纯轮询 | 不推荐 | 患者无感知，体验差 |
| 方案二：订阅消息推送 | 备选 | 体验最好，但配置复杂，超出实训范围 |
| **方案三：轮询 + 红点** | **推荐** | 简单可行，体验足够，适合实训 |

---

## 六、可扩展性

如果后续项目需要升级为实时推送，可以在方案三基础上：

1. 保留轮询 + 红点作为兜底。
2. 医生保存建议后，后端调用微信订阅消息 API 发送推送。
3. 患者点击订阅消息直接进入建议详情页。

这样从“轮询提示”升级到“实时推送”成本很低，只需在后端增加调用微信 API 的逻辑。
