## 二、实训概述

### 2.1 项目目标

SleepCare 是一款面向睡眠健康管理的移动端软件系统，目标是为用户提供一个集睡眠数据采集、睡眠质量评估、睡眠报告展示、医生授权管理与个性化睡眠改善建议于一体的智能健康管理平台。在无硬件依赖的前提下，通过虚拟设备、模拟数据与预留控制接口，完成软件系统的全部业务功能，并为后续真实硬件接入做好准备。

### 2.2 核心功能范围

本项目实现了 9 大功能模块：

1. **用户账号体系**：手机号注册/登录、JWT 鉴权、个人信息管理
2. **设备管理**：虚拟设备添加、设备列表、在线状态、解绑
3. **睡眠报告**：每日报告、睡眠分期柱状图、噪音折线图、趋势图、完整报告
4. **医生授权管理**：授权医生、有效期设置、数据范围、撤销授权、医生端查看报告、医生干预建议
5. **用户设置**：作息计划、助眠偏好、环境设置、设备设置、勿扰时段
6. **e-CBTi 睡眠训练专区**（规划）：睡眠卫生教育、睡眠日记、专业测评、睡眠限制/刺激控制打卡
7. **消息通知**（规划）：报告生成、设备离线、授权提醒
8. **OTA 固件升级**（规划）：版本检查、升级模拟、红点提示
9. **医生端 Web 后台**：医生登录、患者列表、报告详情、干预建议、日志审计

### 2.3 技术栈概述

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | 原生微信小程序 | WXML + WXSS + JS，View 组件绘制图表 |
| 后端 | Node.js + Express | RESTful API 服务 |
| 数据库 | SQLite（本地）/ MySQL（线上） | sql.js 与 mysql2 双模式切换 |
| 认证 | JWT + bcryptjs | Token 鉴权、密码加密 |
| 部署 | 腾讯云 CloudBase + CloudBase Run | 云托管容器 + MySQL + HTTP 网关 |
| 版本控制 | Git + GitHub | 标签 `v1.0-final` 标记最终交付 |

## 三、实训过程中遇到的问题与解决过程

### 3.1 CloudBase 部署：MySQL 内网地址不可达

**问题**：
CloudBase 云托管服务启动后，日志显示 `Error: connect ETIMEDOUT 172.17.0.2:3306`，后端无法连接 MySQL。

**原因**：
CloudBase Run 容器与 MySQL 内网地址不在同一 VPC 内网可达范围内。小租户 MySQL 默认内网地址 `172.17.0.2:3306` 对云托管容器不可见。

**解决**：
在 CloudBase 控制台为 MySQL 实例开启**外网地址**（`sh-cynosdbmysql-grp-0kkqdf2y.sql.tencentcdb.com:23088`），并在云托管服务环境变量中使用该外网地址。同时配置 `charset: 'utf8mb4'`、`timezone: '+08:00'`、`connectTimeout: 30000`、`enableKeepAlive: true` 等连接池参数，增加 5 次重试机制以应对 MySQL 自动暂停冷启动。

### 3.2 SQLite 与 MySQL 异步兼容性问题

**问题**：
切换到 MySQL 后，注册接口报 `Cannot read properties of undefined (reading 'values')`。

**原因**：
SQLite 使用 sql.js 时 `db.exec()` 同步返回数组；MySQL 使用 `mysql2/promise` 时返回 Promise。项目中原有大量 `db.exec()` 调用未加 `await`。

**解决**：
批量将所有 `db.exec()` / `db.run()` 调用改为 `await db.exec()` / `await db.run()`；将 `doctor.js` 中同步函数 `getLatestScore` 改为 `async`；将 `.map()` 循环改为 `for...of` 以支持 `await`。同时为 MySQL 包装器增加空结果判断，确保空查询返回 `[]` 与 sql.js 行为一致。

### 3.3 SQLite 语法与 MySQL 不兼容

**问题**：
医生授权、用户设置等接口返回"授权操作失败"。

**原因**：
代码中使用了 `datetime('now','localtime')` 这种 SQLite 特有语法，以及 `SELECT last_insert_rowid()`，MySQL 不支持。

**解决**：
将所有时间字段改为在 JS 中生成 ISO 时间字符串后作为参数传入 SQL；将 `SELECT last_insert_rowid()` 替换为 `SELECT LAST_INSERT_ID()`，同时保留 SQLite 的 fallback 判断。

### 3.4 医生端 Web 中文乱码

**问题**：
浏览器访问 `doctor.html` 时，中文显示为乱码。

**原因**：
Express 的 `express.static` 中间件未设置 HTML 文件的 `charset` 响应头，浏览器默认按其他编码解析。

**解决**：
在 `app.js` 中为 `express.static` 添加 `setHeaders` 配置，强制 `.html` 文件响应 `Content-Type: text/html; charset=utf-8`。

### 3.5 CloudBase 服务 Liveness 探针失败

**问题**：
部署后 CloudBase 健康检查失败，服务反复重启，报 503。

**原因**：
`initDatabase()` 在 `app.listen()` 之前执行，MySQL 重试 5 次（每次 30 秒）期间服务未监听端口，探针超时。

**解决**：
调整启动顺序：先 `app.listen()` 启动 HTTP 服务，再异步执行 `initDatabase()`；数据库初始化失败时不调用 `process.exit(1)`，改为记录状态但保持服务可用。同时设置最小实例数为 1 避免冷启动 503。

### 3.6 报告页趋势图日期点左偏

**问题**：
报告页趋势图上的日期点/柱状全部偏到左侧，没有对齐到各自日期列。

**原因**：
`.t-dot` 和 `.t-bar` 使用 `transform: translateX(-50%)` 居中但缺少 `left: 50%`，且父容器 `.t-dot-col` 未设置 `position: relative`，导致绝对定位参照整个图表。

**解决**：
为 `.t-dot` 和 `.t-bar` 添加 `left: 50%`，为 `.t-dot-col` 添加 `position: relative`，使其在各自列内居中。

### 3.7 CloudBase 免费额度耗尽导致服务被隔离

**问题**：
部署成功后运行一段时间，MySQL 与云托管资源点超额，服务返回 `SERVICE_FORBIDDEN` / `Your server is isolated`。

**原因**：
免费体验版资源额度为 3000 点，实际消耗 5445.49 点，MySQL 容量使用了 573 MB·时。

**解决**：
为完成作业演示，切换为本地环境运行：将小程序 `baseUrl` 改回局域网 IP `http://10.162.232.136:3000`，本地启动后端 SQLite 模式，确保所有功能可正常演示。

## 四、实训收获与思考（重点）

### 4.1 对 Vibe Coding 的体会

在本次实训中，我通过 AI 辅助完成了大量代码生成、调试与部署工作。Vibe Coding 的核心价值在于：

- **快速原型**：从需求到可运行的 API 和小程序页面，可以在很短时间内完成雏形。
- **跨越知识盲区**：在没有 CloudBase 部署经验的情况下，AI 能引导我逐步完成 MySQL 配置、HTTP 网关、环境变量等操作。
- **持续迭代**：每次报错后，把错误信息直接反馈给 AI，可以高效地定位根因并修复。

但 Vibe Coding 不是"无脑复制"。如果完全依赖 AI，很容易出现两类问题：一是代码只解决当前报错，但引入新的兼容性问题；二是 AI 对上下文记忆有限，可能在修复 A 问题时把 B 问题改回旧状态。所以我的体会是：AI 是加速器，但开发者仍需理解每一行代码的作用。

### 4.2 如何审核 AI 生成的代码

经过本次实践，我总结了几条审核 AI 生成代码的原则：

1. **先读再复制**：把 AI 给出的代码完整看一遍，确认它改的是预期文件、预期位置，而不是无关文件。
2. **检查一致性**：特别是 SQLite 与 MySQL 双模式切换时，任何数据库操作都要确认是否兼容两种数据库。
3. **验证关键路径**：注册、登录、JWT、数据库写操作等核心链路必须手动测试，不能只看 AI 说"已修复"。
4. **保留回退能力**：每次较大修改前先提交 Git，方便回滚。
5. **关注边界情况**：空结果、首次运行、并发请求等场景，AI 往往容易忽略，需要自己补充测试。

### 4.3 整个项目中印象最深的环节及原因

印象最深的是 **CloudBase 部署与 MySQL 兼容问题**。原因在于：

- 这个问题不是代码逻辑错误，而是**环境、网络、数据库方言、容器生命周期**等多因素叠加导致的系统性问题。
- 解决过程中需要同时排查 CloudBase 控制台、Docker 日志、MySQL 连接、Express 启动顺序、SQL 语法兼容性，任何一个环节没解决，整个服务都起不来。
- 它让我真正理解了"本地能跑，线上不一定能跑"的含义，也意识到部署与运维是软件工程中非常重要的一环。

### 4.4 如果再做类似项目，哪些地方会做改进

1. **前期统一数据库抽象层**：如果一开始就把 `db.exec()` 设计成完全异步、对 SQLite 与 MySQL 行为一致的接口，后续切换数据库时就不会出现大量 `await` 补全工作。
2. **更早引入 CI/CD 与自动化测试**：在本地部署到云之前，先跑通集成测试，可以显著减少线上反复调试的次数。
3. **环境配置与代码分离**：把数据库类型、连接信息、域名等完全交给环境变量，避免为本地/线上写两套分支代码。
4. **云资源成本管理**：上线前评估资源消耗，避免因免费额度耗尽导致服务被隔离；必要时使用更低配的实例或定时暂停。
5. **文档先行**：把部署步骤、踩坑记录、环境变量说明写成文档，而不是只依赖 AI 对话历史，对后续复现和交接更有帮助。
